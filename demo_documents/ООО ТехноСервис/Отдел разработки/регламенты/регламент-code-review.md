# Регламент РЕГ-ДЕВ-001

## Регламент проведения code review

### 1. Цель

Обеспечение качества кода и соответствия стандартам разработки через обязательный просмотр всех изменений кода перед их слиянием в основную ветку.

### 2. Область применения

Регламент распространяется на все изменения кода в репозиториях организации.

### 3. Процесс code review

3.1. **Создание Pull Request**
   - Разработчик создает Pull Request (PR) в системе контроля версий
   - PR должен содержать описание изменений
   - PR должен быть связан с задачей в системе учета задач

3.2. **Назначение ревьюера**
   - Ревьюер назначается автоматически или вручную
   - Ревьюером может быть любой разработчик, кроме автора изменений
   - Для критических изменений требуется минимум 2 ревьюера

3.3. **Проведение review**
   - Ревьюер проверяет код на соответствие стандартам
   - Проверяется логика работы кода
   - Проверяется наличие тестов
   - Проверяется документация

3.4. **Результат review**
   - **Approve** - код одобрен, можно мержить
   - **Request changes** - требуются изменения
   - **Comment** - комментарий без блокировки

### 4. Критерии оценки

4.1. Код должен быть читаемым и понятным.

4.2. Код должен соответствовать принятым стандартам стиля.

4.3. Должны быть написаны тесты для новой функциональности.

4.4. Не должно быть закомментированного кода.

4.5. Не должно быть "магических чисел" и строк.

### 5. Сроки

5.1. Code review должен быть проведен в течение 24 часов.

5.2. Критичные изменения должны быть рассмотрены в течение 4 часов.

5.3. Автор PR должен ответить на комментарии в течение 8 часов.

### 6. Ответственность

6.1. Разработчик несет ответственность за качество своего кода.

6.2. Ревьюер несет ответственность за тщательность проверки.

6.3. Руководитель отдела контролирует соблюдение регламента.
